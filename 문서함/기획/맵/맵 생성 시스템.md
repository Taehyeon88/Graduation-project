# 1. 시스템 개요

완전 무작위 생성의 단점을 방지하고 의도가 반영된 전술적 환경을 제공하는 것을 목표로 함. ‘청크 패턴’으로 멥의 뼈대를 잡고, ‘가중치 기반 주머니 추첨’으로 남은 빈칸을 채워 통제된 무작위성을 가진 전술 전투 맵을 생성함.

# 2. 핵심 개념 및 작동 원리

상세 프로세스 진입 전, 이 시스템이 어떻게 '전술적으로 의미 있는 랜덤 맵'을 만들어내는지 4가지 핵심 원리를 먼저 정의함.

### **청크 패턴 - '미리 만들어둔 레고 블록 도장 찍기'**

맵의 49칸을 완전히 무작위로 하나씩 채우면, 벽과 함정이 지저분하게 흩뿌려져 전술적인 재미가 사라짐. 

이를 방지하기 위해 미리 '일자형 바리케이드', '지그재그 벽' 같은 2x2, 3x2 크기의 의미 있는 덩어리(청크)를 만들어 둠. 

맵 생성 초반에 이 덩어리들을 빈 맵 위에 도장 찍듯 통째로 툭툭 배치하여, 랜덤 맵 안에서도 의도한 최소한의 방어선이나 병목 구간(뼈대)이 확정적으로 만들어지게 함.

### **데이터 주도형 테마**

맵 생성기는 고정된 알고리즘으로만 작동하되, 외부에서 입력 받는 `Theme_ID` 에 따라 맵에 찍히는 ‘청크 패턴의 종류’, ‘장애물의 속성’. ‘등장 확률(가중치)’이 실시간으로 교체되어 수십 가지 컨셉의 맵을 찍어 낼 수 있음.

### 동적 제한 시스템

맵 전체의 방해물 개수와 개별 오브젝트의 설치 한도를 실시간으로 추적함.

생성 도중 한도에 도달한 오브젝트는 스폰 가중치가 즉시 `0`으로 잠기며, 확률 주머니에서 영구 제외되어 남은 타일들에게 스폰 확률이 자동으로 재분배됨.

### 역할의 완전 분리

본 시스템은 오직 그리드 위의 지형 및 오브젝트 배치와 이동 경로 검증만 담당함.

몬스터 스폰 및 웨이브 연산은 본 시스템이 제공한 ‘유효 평지 데이터’를 받아 외부 전담 시스템에서 독립적으로 수행함. 

# 3. 상세 프로세스

맵 생성은 총 6단계의 순서로 진행됨.

## 단계 1 : 테마 데이터 로드

진입할 스테이지의 `Theme_Id`를 확인하고, 해당 테마에 종속된 ‘사용 패턴 풀(Pattern_Pool)’과 '오브젝트 스폰 가중치(Object_Spawn_Data)'를 메모리에 로드함.

![image.png](image.png)

## 단계 2 : 초기화 및 안전 구역 마스킹

7x7 크기의 빈 배열을 생성(모든 칸 EMPTY(0))함.

플레이어 시작점(0, 3/0, 4/0, 5) 고정하고, `MASKED`(배치 불가) 상태로 잠가 덮어쓰기를 원천 차단함.

![image.png](image%201.png)

## 단계 3 : 청크 데이터 배치

1~3회의 난수를 뽑아 뼈대 생성 횟수를 정함.

테마에 지정된 ‘청크 데이터’을 무작위로 선택하여 맵의 유효한 무작위 좌표에 덮어씌움.

단, 청크 데이터의 빈칸(0)과 맵의 `MASKED` 좌표는 덮어쓰지 않고 무시함.

## 단계 4 : 동적 제한 세팅

테마의 오브젝트 총 허용치(Global_Obstacle) 범위 내에서 이번 맵의 ‘오브젝트 최대 한도'를 난수로 결정함.

Phase 2에서 맵에 미리 깔린 방해물의 총개수를 세어 '남은 전체 여유분'을 계산하고, 미리 깔린 개별 타일의 개수 역시 `Current_Count`에 선반영하여 동기화함.

## 단계 5 : 테마 기반 빈칸 채우기

맵에 남은 빈칸 수만큼 테마 스폰 가중치를 기반으로 '룰렛 휠 셀렉션' 추첨을 반복함.

개별 타일의 스폰 카운트가 `max_Limit`에 도달하거나, 오브젝트 전체 여유분이 `0`이 되면 해당 그룹의 가중치를 일괄 `0`으로 잠금.

완성된 타일 주머니를 섞어(Shuffle) 맵의 빈칸에 차례대로 배치함.

## 단계 6 : 길 찾기 검증 및 시스템 인계

4방향 BFS를 통해 플레이어 시작점에서 목표 지점까지 경로가 이어지는지 검증.

빈 타일을 전부 이동이 가능한 상태여야 함.

- 성공 시
    - 생성된 맵에 몬스터 스폰 시스템에 인계하여 몬스터 배치
- 실패 시
    - 즉시 맵을 파괴하고 단계 2로 돌아갑니다.

# 4. 세부 규칙

### 맵 생성 기본 조건

맵 크기는 7x7 그리드 타일을 사용하며, 생성 시스템은 몬스터 스폰 데이터와 완전히 분리되어 지형과 오브젝트 배치만을 담당함.

### 오브젝트 속성 분류

- Empty : 빈칸
- Unbr : 파괴 불가 오브젝트
- BR : 파괴 가능 오브젝트
- Trap : 함정

### BFS 길 찾기 검증

- 목적 : 맵이 장애물로 완정히 막혀 클리어가 불가능을 방지.
- 통과 조건 : Is_Passable = true 속성을 가진 타일만 이동 가능한 경로로 인정함.

# 5. 밸런싱 데이터 구조

### object_Base_Table

- object_Id : 시스템 및 타 테이블에서 오브젝트를 식별하기 위한 고유 정수값.
- name : 오브젝트를 쉽게 인지하기 위한 명칭.
- type : 오브젝트의 그룹을 규정하는 속성. ‘동적 제한 세팅’에 사용됨.
- heath : 파괴 가능 오브젝트의 체력 표기
- Is_Passable : 맵 검증 알고리즘 수행 시 해당 타일을 통과 가능한 빈 공간으로 취급할지 벽으로 취급할지 결정하는 논리값.
    - True : 통과 가능
    - False : 통과 불가능

### chunk_Table

- chunk_Id : 맵 생성 시 스폰할 청크를 식별하기 위한 고유 정수값.
- chunk_Name : 청크을 나타내는 명칭.
- chunk_Size : 패턴 배열이 차지하는 가로(W)와 세로(H) 타일 칸 수.
- chunk_Data : 실제 패턴의 형태를 구성하는 데이터.

### map_Theme_Table

- Theme_Id : 테마의 고유 식별자.
- Theme_name : 테마의 컨셉을 나타내는 명칭.
- global_Obstacle : 해당 테마 맵 1개에 존재할 수 있는 오브젝트의 총합 제한 수치(최소~최대 난수 범위)
- chunk_Pool : 해당 테마에서 사용할 수 있는 chunk_Id의 목록 (쉼표로 구분)
- object_Spawn_Data : 단계 5 빈칸 채우기 시 사용되는 스폰 룰렛 데이터.
    - 쉼표( , ) : 문자열을 쪼개어 개별 오브젝트 문자열 배열로 만듬.
    - 콜론 ( : ) : [object_Id, spawn_Weight, max_Spawn_Limit] 3개의 인자가 파싱되어 연산
        - object_Id : 맵에 배치될 오브젝트 고유 Id
        - spawn_Weight : 해당 오브젝트의 생성 가중치
        - max_Spawn_Limit : 맵에 배치될 수 있는 최대 수량

# 6. 예외 처리 방안

### 예외 1 : 청크 패턴 투하로 인한 개별 스폰 최대치 초과 발생

- 상황 : 가시 함정 최대치가 5개이나, 청크 패턴이 여러번 떨어져 이미 배치될 수 있는 최대 수량을 넘겨 생성된 경우
- 처리 : 유연성을 위해 이를 ‘정상적인 돌발 변수’로 허용(맵에서 삭제하지 않음)함. 단, 빈칸 채우기 진입 시 해당 타일의 현재 스폰 카운트가 이미 최대치를 초과하였으므로 스폰 가중치를 즉시 0으로 잠가 추가 스폰을 차단.
